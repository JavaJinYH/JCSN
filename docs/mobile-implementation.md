## 一、项目概述

### 1.1 目标
为折柳建材管理系统添加移动端适配，采用**混合架构**：
- **电脑端**：保持现有功能不变（继续用 Electron）
- **手机端**：Capacitor 打包，只读缓存（不修改数据），断网也能看旧数据
- **数据同步**：通过 Syncthing 在局域网同步照片目录（数据库不用同步！）

### 1.2 目标用户

| 用户 | 设备 | 场景 |
|------|------|------|
| 爸爸/妈妈 | 各持有一台手机/平板 | 查看库存、订单，拍照上传签字单 |

**注意**：送货员不接入系统，签字单照片通过微信/电话发给老板，由老板上传到系统。

### 1.3 核心原则

1. **同一套代码** - 电脑端和手机端共用 95% 以上代码
2. **不影响电脑端** - 继续优化和调试电脑端，互不影响
3. **渐进式开发** - 先做响应式，再加新功能
4. **离线查看旧数据** - 手机端有只读缓存，断网也能看旧数据
5. **不冲突原则** - 手机端不修改数据，完全避免同步冲突

### 1.4 多设备场景

**家里两台手机 + 店里一台电脑**，数据实时同步：

```
┌─────────────────┐        WiFi         ┌─────────────────┐
│  爸爸的手机（只读缓存）│ ◄────────────► │  店里电脑（唯一权威）│
│  - 在局域网时：看最新数据 │                    │  - SQLite 数据库     │
│  - 不在局域网时：看旧数据 │                    │  - Photos 目录      │
│  - 拍照本地缓存（等网了再上传 │                    │  - Syncthing        │
├─────────────────┤                    │  - 局域网 API 服务  │
│  妈妈的手机（只读缓存）│ ◄────────────► │                     │
│  - 在局域网时：看最新数据 │                    │                     │
│  - 不在局域网时：看旧数据 │                    │                     │
│  - 拍照本地缓存（等网了再上传 │                    │                     │
└─────────────────┘                    └─────────────────┘
         │
         ▼

```

**关键点**：
1. **仅家庭成员使用** - 不对外开放，保证数据安全
2. **多台手机可同时使用** - 均通过局域网 API 上传照片到主电脑
3. **店里电脑是唯一权威** - 手机端只读，不修改数据，完全避免冲突
4. **照片上传统一到主电脑** - 所有照片存在主电脑的 Photos 目录
5. **待处理单据实时更新** - 一台手机上传后，其他设备刷新可见
6. **离线查看旧数据** - 手机不在局域网也能看最后一次缓存的旧数据，提示"数据可能不是最新"

## 二、技术方案（混合架构，无冲突）

### 2.1 核心原则

| 原则 | 说明 |
|------|------|
| 📦 **同一套代码** | 电脑端和手机端共用 95% 以上代码 |
| 🛡️ **不影响电脑端** | 继续优化调试，互不影响 |
| 🏗️ **渐进式开发** | 先做响应式，再加新功能 |
| 📴 **离线查看旧数据** | 手机端有只读缓存，断网也能看旧数据 |
| ✅ **完全避免冲突** | 手机端只读，不修改数据 |

### 2.2 技术栈

| 部分 | 技术 | 说明 |
|------|------|------|
| **电脑端** | Electron + React + Prisma + SQLite | 保持不变，继续用 |
| **手机端** | Capacitor + React + LocalStorage（只读缓存） | 本地缓存，离线可用 |
| **打包工具** | Electron Builder（电脑端）/ Capacitor（手机端） | 分开打包 |
| **局域网服务** | Express（集成到 Electron 主进程） | 不单独建 server 目录 |
| **数据同步** | Syncthing（只同步 Photos 目录） | 数据库不用同步，避免冲突 |
| **相册/摄像头** | 标准 Web API | 无需适配不同手机型号 |

## 五、常见问题解答

### Q1：会不会影响我现在优化电脑端？
**完全不会！**：电脑端功能完全不变，你可以继续优化调试，互不影响。

### Q2：是不是要开发一套新系统？
**不是！** 95% 的代码继续用，只是加个响应式和底部导航。

### Q3：两台手机能不能同时用？
**可以！** 主电脑开启局域网服务，两台手机都能访问。

### Q4：数据怎么同步？
用 Syncthing 只同步 Photos 目录（数据库不用同步，避免冲突！）。

### Q5：手机不在局域网能看数据吗？
**可以！** 显示最后一次在局域网时缓存的旧数据，同时提示"⚠️ 不在店里 WiFi 网络，数据可能不是最新"。

### Q6：手机不在局域网能拍照上传吗？
**可以！** 照片会保存到本地缓存队列，等回到局域网了自动排队上传。

### Q7：店里电脑修改了订单，手机回来会有冲突吗？
**完全不会！** 手机端只读不写，每次在局域网都是直接调用 API 看最新数据，完全没问题。

### Q8：两个人都出门了，都拍了照回来上传，会冲突吗？
**不会！** 照片排队上传，按时间顺序处理，完全没问题。

## 七、Git 提交说明

本项目使用 **明确的提交类型**，便于区分电脑端和移动端的改动：

| 提交前缀 | 说明 |
|---------|------|
| `feat(mobile):` | 📱 移动端新功能 |
| `fix(mobile):` | 🐛 移动端 Bug 修复 |
| `refactor(mobile):` | ♻️ 移动端重构 |
| `style(mobile):` | 🎨 移动端样式 |
| `docs(mobile):` | 📚 移动端文档 |
| `feat(desktop):` | 💻 电脑端新功能 |
| `fix(desktop):` | 🐛 电脑端 Bug 修复 |
| `refactor(desktop):` | ♻️ 电脑端重构 |
| `docs(desktop):` | 📚 电脑端文档 |

**示例提交：**
```bash
# 移动端提交
git add docs/mobile-implementation.md
git commit -m "docs(mobile): 更新移动端实施进度"

# 电脑端提交
git commit -m "feat(desktop): 添加新的销售报表功能"
```

这样以后回滚代码时，就不会分不清哪些是电脑端改动，哪些是移动端改动了！
