# 架构重构方案：联系人-结账主体模型

## 一、设计背景

### 1.1 现有架构问题

当前 `Customer` 表混合了"人"和"付款主体"两个概念：
- 手机号可能重复创建
- 付款主体和个人关系不清晰
- 账目挂在谁身上不清楚
- 无法支持"谁买东西 ≠ 谁付钱"的业务场景

### 1.2 核心设计原则

1. **人是独立的**：一个人只录一次，用手机号唯一标识
2. **身份是临时的**：一个人可以同时拥有多个身份、参与多个项目
3. **买付分离**：谁买东西 ≠ 谁付钱，付款方是独立的实体
4. **账目清晰**：所有账目都挂在"付款主体"下，而不是挂在"采购人"下

## 二、新数据结构（6表设计）

### 2.1 contacts（联系人/人员）

一个人只有一个记录，手机号唯一。

```prisma
model Contact {
  id           String   @id @default(cuid())
  name         String
  primaryPhone String   // 主手机号，唯一
  address      String?
  remark       String?
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  // 关联
  phones           ContactPhone[]       // 多手机号
  entityRoles      ContactEntityRole[]  // 在各主体的角色
  projectRoles     ContactProjectRole[]  // 在各项目的角色
  ordersAsBuyer    Order[]              @relation("Buyer")
  ordersAsPayer    Order[]              @relation("Payer")
  ordersAsIntroducer Order[]             @relation("Introducer")
  ordersAsPicker   Order[]              @relation("Picker")
}
```

### 2.2 ContactPhone（联系人手机号）

支持一个人有多个手机号。

```prisma
model ContactPhone {
  id          String   @id @default(cuid())
  contactId   String
  contact     Contact  @relation(fields: [contactId], references: [id], onDelete: Cascade)
  phone       String
  isPrimary   Boolean  @default(false)
  createdAt   DateTime @default(now())

  @@unique([contactId, phone])
}
```

### 2.3 entities（结账主体）

可以是个人（用联系人表示）或公司/单位。

```prisma
model Entity {
  id           String   @id @default(cuid())
  name         String
  entityType   String   // "personal" | "company" | "government"
  contactId    String?  // 如果是个人，对应的联系人
  contact      Contact? @relation(fields: [contactId], references: [id])
  defaultPayer String?  // 默认付款人
  creditLimit  Float    @default(0)
  creditUsed  Float    @default(0)
  address      String?
  remark       String?
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  // 关联
  roles       ContactEntityRole[]
  orders      Order[]
}
```

### 2.4 projects（项目）

```prisma
model Project {
  id           String   @id @default(cuid())
  name         String
  entityId     String   // 项目所属的结账主体
  entity       Entity   @relation(fields: [entityId], references: [id], onDelete: Cascade)
  address      String?
  status       String   @default("进行中") // 进行中/已完成/已暂停/已取消
  startDate    DateTime?
  endDate      DateTime?
  estimatedAmount Float?
  remark       String?
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  // 关联
  roles       ContactProjectRole[]
  orders      Order[]
}
```

### 2.5 contact_entity_roles（联系人-主体角色）

定义一个人在某个主体下的角色。

```prisma
model ContactEntityRole {
  id          String   @id @default(cuid())
  contactId   String
  contact     Contact  @relation(fields: [contactId], references: [id], onDelete: Cascade)
  entityId    String
  entity      Entity   @relation(fields: [entityId], references: [id], onDelete: Cascade)
  role        String   // "buyer" | "payer" | "introducer" | "supervisor"
  isDefault   Boolean  @default(false) // 是否为该主体的默认角色
  remark      String?
  createdAt   DateTime @default(now())

  @@unique([contactId, entityId, role])
}
```

### 2.6 contact_project_roles（联系人-项目角色）

定义一个人在某个项目下的角色。

```prisma
model ContactProjectRole {
  id          String   @id @default(cuid())
  contactId   String
  contact     Contact  @relation(fields: [contactId], references: [id], onDelete: Cascade)
  projectId   String
  project     Project  @relation(fields: [projectId], references: [id], onDelete: Cascade)
  role        String   // "buyer" | "payer" | "introducer" | "manager"
  remark      String?
  createdAt   DateTime @default(now())

  @@unique([contactId, projectId, role])
}
```

### 2.7 orders（订单/销售单）

核心业务表，所有账目挂在 `payment_entity_id` 下。

```prisma
model Order {
  id               String   @id @default(cuid())
  invoiceNo        String?
  writtenInvoiceNo String?
  saleDate         DateTime @default(now())

  // 三种角色关联（都指向Contact）
  buyerId          String
  buyer            Contact  @relation("Buyer", fields: [buyerId], references: [id])

  introducerId     String?  // 介绍人（水电工）
  introducer       Contact? @relation("Introducer", fields: [introducerId], references: [id])

  pickerId         String?  // 提货人
  picker           Contact? @relation("Picker", fields: [pickerId], references: [id])
  pickerName       String?
  pickerPhone      String?

  // 项目关联
  projectId        String?
  project          Project? @relation(fields: [projectId], references: [id])

  // 付款主体（核心：所有账目挂在这里）
  paymentEntityId  String
  paymentEntity     Entity   @relation(fields: [paymentEntityId], references: [id])

  // 金额
  totalAmount      Float
  discount         Float    @default(0)
  paidAmount       Float    @default(0)
  writeOffAmount   Float    @default(0)

  // 状态
  status           String   @default("completed") // pending/completed/partial/cancelled

  remark           String?
  createdAt        DateTime @default(now())
  updatedAt        DateTime @updatedAt

  // 关联
  items            OrderItem[]
  payments         Payment[]
  receivables      Receivable[]
}
```

## 三、场景测试用例

### 场景1：水电工张三自己家买东西，自己付钱

```javascript
// 创建联系人
张三 = Contact.create({ name: "张三", primaryPhone: "13800138001" })

// 创建个人结账主体（张三自己就是付款主体）
张三主体 = Entity.create({
  name: "张三",
  entityType: "personal",
  contactId: 张三.id
})

// 无项目，直接开单
Order.create({
  buyerId: 张三.id,
  paymentEntityId: 张三主体.id,
  totalAmount: 1000,
  paidAmount: 0  // 欠款挂在这里
})
```

### 场景2：张三作为A装修公司采购员，为A公司的X项目买东西，A公司付钱

```javascript
// 联系人
张三 = Contact.create({ name: "张三", primaryPhone: "13800138001" })
A公司 = Entity.create({ name: "A装修公司", entityType: "company" })

// 项目
X项目 = Project.create({
  name: "龙湖天街水电工程",
  entityId: A公司.id
})

// 角色关联
ContactEntityRole.create({
  contactId: 张三.id,
  entityId: A公司.id,
  role: "buyer"
})

// 开单
Order.create({
  buyerId: 张三.id,
  projectId: X项目.id,
  paymentEntityId: A公司.id,  // 账挂在A公司
  totalAmount: 5000
})
```

### 场景3：张三为B业主家的Y项目买东西，业主把钱给张三，张三付钱

```javascript
// 联系人
张三 = Contact.create({ name: "张三", primaryPhone: "13800138001" })
B业主 = Contact.create({ name: "B业主", primaryPhone: "13900139001" })

// B业主的个人结账主体
B业主主体 = Entity.create({
  name: "B业主",
  entityType: "personal",
  contactId: B业主.id
})

// 项目
Y项目 = Project.create({
  name: "B业主家装修",
  entityId: B业主主体.id
})

// 张三在项目中的角色是采购
ContactProjectRole.create({
  contactId: 张三.id,
  projectId: Y项目.id,
  role: "buyer"
})

// 开单 - 付款主体是B业主主体
Order.create({
  buyerId: 张三.id,           // 张三采购
  projectId: Y项目.id,
  paymentEntityId: B业主主体.id,  // 钱实际上是B业主的
  totalAmount: 3000
})
```

### 场景4：张三为B业主家的Y项目买东西，业主直接付款

```javascript
// 联系人
B业主 = Contact.create({ name: "B业主", primaryPhone: "13900139001" })

// B业主主体
B业主主体 = Entity.create({
  name: "B业主",
  entityType: "personal",
  contactId: B业主.id
})

// 项目
Y项目 = Project.create({
  name: "B业主家装修",
  entityId: B业主主体.id
})

// 开单 - B业主既是采购人也是付款人
Order.create({
  buyerId: B业主.id,
  projectId: Y项目.id,
  paymentEntityId: B业主主体.id,
  totalAmount: 2000
})
```

### 场景5：张三同时在3个项目干活，分别为3个不同的付款主体买东西

```javascript
// 联系人
张三 = Contact.create({ name: "张三", primaryPhone: "13800138001" })

// 3个结账主体
主体A = Entity.create({ name: "A公司", entityType: "company" })
主体B = Entity.create({ name: "B公司", entityType: "company" })
主体C = Entity.create({ name: "C业主", entityType: "personal" })

// 3个项目
项目1 = Project.create({ name: "A公司项目", entityId: 主体A.id })
项目2 = Project.create({ name: "B公司项目", entityId: 主体B.id })
项目3 = Project.create({ name: "C业主项目", entityId: 主体C.id })

// 张三在3个项目都是采购角色
ContactProjectRole.createMany([
  { contactId: 张三.id, projectId: 项目1.id, role: "buyer" },
  { contactId: 张三.id, projectId: 项目2.id, role: "buyer" },
  { contactId: 张三.id, projectId: 项目3.id, role: "buyer" },
])

// 3笔订单，账分别挂在3个主体
Order.create({ buyerId: 张三.id, projectId: 项目1.id, paymentEntityId: 主体A.id, ... })
Order.create({ buyerId: 张三.id, projectId: 项目2.id, paymentEntityId: 主体B.id, ... })
Order.create({ buyerId: 张三.id, projectId: 项目3.id, paymentEntityId: 主体C.id, ... })
```

## 四、数据迁移方案

### 4.1 迁移原则

1. **保持兼容性**：保留原有的 Customer 表，但标记为旧数据
2. **双写策略**：新旧系统并行运行，逐步迁移
3. **数据验证**：迁移后进行数据一致性校验

### 4.2 迁移步骤

#### 步骤1：创建新表结构

```bash
npx prisma migrate dev --name add_contact_entity_model
```

#### 步骤2：数据迁移脚本

1. 将现有 Customer 迁移到 Contact + Entity（个人）
2. 将 CustomerCategory 迁移到 Entity（公司类型）
3. 迁移 Project 关联
4. 迁移 Sale 到 Order
5. 迁移关联关系

#### 步骤3：验证数据

- 检查手机号唯一性
- 检查账目金额一致性
- 检查关联关系完整性

### 4.3 回滚方案

如果迁移失败，可以：
1. 删除新表结构
2. 恢复旧表数据
3. 使用备份恢复

## 七、注意事项

1. **手机号唯一性**：必须在数据库层面和业务逻辑层双重校验
2. **付款主体必填**：开单时 paymentEntityId 不能为空
3. **账目清晰**：所有财务统计都基于 paymentEntity
4. **兼容旧数据**：迁移期间保持 Customer 表可用
